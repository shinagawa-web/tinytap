# eBPF Basics: How C, the Kernel, and Go Fit Together

This document explains the foundational concepts behind tinytap's architecture.

## The Three-Layer Structure

```
┌─────────────────────────────────────┐
│  User Space (Go)                    │
│                                     │
│  main.go                            │
│    └─ LoadTinytapObjects()          │  ← generated by bpf2go
│         └─ loads eBPF into kernel   │
│                                     │
│  reads captured data from maps      │
└────────────────┬────────────────────┘
                 │ syscall (bpf())
┌────────────────▼────────────────────┐
│  Kernel Space (eBPF VM)             │
│                                     │
│  tinytap.bpf.c runs here            │  ← written in C
│    └─ hooks into network events     │
│         └─ writes data to maps      │
└─────────────────────────────────────┘
```

## What eBPF Is

eBPF is a sandboxed virtual machine that runs safely inside the Linux kernel.

Traditionally, extending kernel behavior required writing a kernel module (`.ko`). A bug in a module can crash the entire system. eBPF solves this:

- **Traditional kernel module**: a bug causes a kernel panic — dangerous
- **eBPF program**: the kernel's verifier checks the program before loading it — safe

tinytap uses eBPF to hook into network events and capture HTTP traffic without modifying the kernel itself.

## The C Program: Hooking Into the Kernel

eBPF programs are written in C and compiled to eBPF bytecode. The `SEC()` macro declares where in the kernel the program attaches:

```c
SEC("kprobe/tcp_sendmsg")
int handle_tcp_send(struct pt_regs *ctx) {
    // this runs inside the kernel every time tcp_sendmsg is called
    return 0;
}

char LICENSE[] SEC("license") = "GPL";
```

Common attachment points:

| `SEC(...)` value          | Where it attaches                          |
|---------------------------|--------------------------------------------|
| `"kprobe/tcp_sendmsg"`    | entry of the `tcp_sendmsg` kernel function |
| `"xdp"`                   | the moment a packet arrives at the NIC     |
| `"tracepoint/..."`        | a kernel tracepoint                        |
| `"socket"`                | socket-level packet filter                 |

## Maps: Sharing Data Between Kernel and Go

Data captured in the kernel needs to be passed to user space. **eBPF maps** are the bridge — shared memory accessible from both sides.

```
Kernel (C)                       User Space (Go)
┌──────────────────┐             ┌──────────────────┐
│ eBPF program     │             │ main.go           │
│                  │◄───────────►│                  │
│ writes data      │  shared map │ reads data        │
│ into map         │             │ from map          │
└──────────────────┘             └──────────────────┘
```

Defining a map in C:

```c
struct {
    __uint(type, BPF_MAP_TYPE_HASH);
    __uint(max_entries, 1024);
    __type(key, int);
    __type(value, char[256]);
} events SEC(".maps");
```

## What bpf2go Does

Go cannot directly use `.o` files, so `bpf2go` acts as the bridge:

```
tinytap.bpf.c
    ↓  clang  (invoked by go generate)
tinytap_bpfel.o    ← compiled eBPF bytecode (ELF)
    ↓  bpf2go
tinytap_bpfel.go   ← embeds the .o file + generates loader functions
```

`tinytap_bpfel.go` is entirely auto-generated. You never edit it by hand — just run `go generate` after changing the C source.

## eBPF Is Not Ordinary C

Because eBPF programs run inside the kernel, there are strict constraints:

| Constraint                    | Reason                                          |
|-------------------------------|-------------------------------------------------|
| Loops must be bounded         | prevents infinite loops from freezing the kernel |
| No `malloc`                   | no dynamic memory allocation                    |
| Only kernel helper functions  | arbitrary function calls are not allowed        |
| Stack limited to 512 bytes    | small, fixed stack size                         |
| No `printf`                   | use `bpf_printk()` for debug output instead     |

## The Plan for tinytap

```
1. A packet arrives at the NIC
         ↓
2. The eBPF program (C) runs
   - extracts the TCP payload
   - looks for "GET /" or "HTTP/1.1"
   - writes the result to a map
         ↓
3. main.go polls the map
   - reads and displays the captured HTTP data
```

## The Current State of tinytap.bpf.c

The C file currently contains only the skeleton:

```c
//go:build ignore

#include <linux/bpf.h>
#include <bpf/bpf_helpers.h>

char LICENSE[] SEC("license") = "GPL";
```

- `//go:build ignore` — tells the Go build system to skip this file; bpf2go compiles it separately with clang
- `#include <linux/bpf.h>` — kernel header with eBPF types, constants, and helper declarations
- `#include <bpf/bpf_helpers.h>` — libbpf header providing `SEC()`, `bpf_printk()`, map helpers, etc.
- `LICENSE` — required; tells the kernel this program is GPL-licensed, which unlocks certain helper functions

The next step is to add a map definition and a program with a `SEC(...)` hook.
