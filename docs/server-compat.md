# Server Compatibility

> Part of v0.4.0 (#37). Manual exploration — each row is filled in by running tinytap against a real server and observing the captured traffic.
>
> Verification criteria reset (this revision) now that #36 (256 B → 4096 B payload cap), #111 (writev/readv multi-iovec sampling), #116 (chunked CRLF-drop no longer abandons), and #117 (binary body TUI placeholder) have all landed. Every server row below — including ones previously filled in — starts over from scratch against the current criteria; nothing from an earlier revision of this doc should be assumed to still hold.

## How to read this table

**Syscall** — the syscall(s) the server uses to send the HTTP response body. Determines whether tinytap can see the body at all, and whether the writev/readv multi-iovec fix (#111) or the still-open sendmsg/recvmsg multi-iovec gap (#113) is in play.

**Pairing** — whether the exchange completes successfully (a status code is shown) or is reported `ABANDONED`. A body being truncated by the sample cap does not, by itself, cause an abandon (#35/#36) — only specific framing bytes being dropped can (#116 fixed the chunked CRLF case; #122 tracks the still-open chunked trailer case).

**Body visibility** — what the TUI shows for the response body:

| Symbol | Meaning |
|--------|---------|
| ✅ | Full body visible |
| ⚠️ | Visible but truncated at the 4096 B BPF cap (#36) — exchange still pairs successfully |
| ❌ | Not captured (sendfile / splice path — body bypasses the BPF probe; the #68 kprobe covers this on arm64 only today, #112 tracks x86_64) |
| 🚫 | Exchange reported `ABANDONED` instead of pairing |
| — | Not yet tested |

**Body sizes / cases used in each run:**

| Label | Size | Rationale |
|-------|------|-----------|
| Small | < 1 KiB | Comfortably fits within the 4096 B sample cap — the "everything visible" baseline |
| Medium | ~8 KiB | Exceeds the cap by roughly 2x — tests truncated-but-paired behavior |
| Large | ~50 KiB | Forces the server to issue multiple write/chunk calls or use sendfile |
| Image | a real image file, `Content-Type: image/*` | Confirms the TUI shows the binary placeholder (#117) instead of a hex/decoded dump — a display check, not a capture-visibility check |

**Additional things to note per server, beyond the visibility table:**

- Does `writev`/`readv` carry the body in a later iovec (not `iovec[0]`)? If so, #111's fix is what makes it visible at all — worth calling out explicitly since it's easy to misattribute visibility to the cap alone.
- Does `sendmsg`/`recvmsg` appear, and with more than one iovec? If so, flag it as affected by #113 (not yet fixed) — body living in `iovec[1+]` of a `sendmsg`/`recvmsg` call is invisible regardless of the cap.
- Is `Transfer-Encoding: chunked` used? Confirm the exchange pairs successfully (#116) rather than `ABANDONED`. If the server sends trailer fields (uncommon), note whether #122's still-open gap is hit.
- Is `sendfile`(2) used for static files? Confirm the arm64 kprobe (#68) still samples a prefix; note that x86_64 has no equivalent yet (#112).
- Any `TCP_CORK` / `MSG_MORE` / kernel buffering surprises worth recording for #36's design history.

## Notes for #36 / #111 / #113 / #116 / #122 (per-server surprises)

> Collect per-server surprises (TCP_CORK, MSG_MORE, kernel buffering, sendfile gaps, multi-iovec placement) here as rows are filled in.

- **Python `http.server` (#41)**: confirmed with `strace -f -e trace=write,sendto,sendmsg,writev,sendfile,recvmsg` against 200 B, 8192 B, 51200 B, and a 68 B PNG. `BaseHTTPRequestHandler`/`SimpleHTTPRequestHandler` always writes headers and body as two separate `sendto` calls, and the body call is a single `sendto` regardless of size — no `sendfile`, no `writev`, no chunking into multiple writes, even for the 50 KiB body. This means the entire body-visibility story for this server is just "first 4096 B of whatever the body is" — there's no per-chunk boundary, no multi-iovec placement (#111/#113 don't apply), and no chunked-encoding pairing behavior to worry about (#116/#122 don't apply either). `mimetypes.guess_type` correctly identifies the PNG fixture as `image/png`, so the #117 TUI placeholder triggers as expected. This server remains the simplest possible baseline — every other server's row should be compared against "does it behave more like Python (plain syscalls, nothing fancy) or does it diverge (sendfile, multi-iovec, chunked, etc.)?"
- **Go `net/http` (#42)**: exercised on this Lima VM (arm64 only — no x86_64 host was available for this pass). Two things were checked independently and are reported separately below: the syscall shape (via `strace -f -e trace=network,write,writev,sendto,sendmsg,sendfile` on the server process) and the actual captured/kept byte count and pairing outcome (via the real `tinytap` TUI, driven non-interactively through `tmux send-keys`/`capture-pane` — `stdout -v` only prints headers, not body samples or the truncated flag, so it can't answer this on its own).
  - **Syscall shape (strace)**: `ServeContent`'s `io.ReaderFrom` fast path behaves differently depending on body size. For small.txt (200 B) and image.png (68 B) — both ≤512 B — the whole response goes out as a single `write` (386 B and 237 B on the wire) and `sendfile` is never called. For medium.txt (8192 B) and large.txt (51200 B) — both >512 B — the same fast path always writes the headers plus exactly the *first 512 B* of the body in one `write` call (699 B and 700 B on the wire), then hands the entire remainder to a single `sendfile` call (7680 B and 50688 B, i.e. `total − 512` in both cases). This 512 B split was consistent across both sizes and happened regardless of whether `Content-Type` was already resolved by the file extension.
  - **Captured bytes / pairing (real TUI, not inferred)**: tinytap captures each HTTP exchange from *two* independent vantage points — the server process's outgoing syscalls and the curl client's incoming syscalls — and pairs each side separately, so there are two detail-panel rows per request. Opening both rows in the actual TUI (repeated twice, same result each time) gave:
    - small.txt: **200/200 B**, not truncated, on both the server row and the client row.
    - medium.txt (8192 B total): server row **4096/8192 B — truncated**; client row **4608/8192 B — truncated** (a second run also produced 4608 for the client row).
    - large.txt (51200 B total): server row **4096/51200 B — truncated**; client row **3908/51200 B — truncated** (reproduced identically on a second run).
    - image.png (68 B total, `Content-Type: image/png`): **`Response body: [image/png, 68 bytes]`** on both rows — the #117 binary placeholder rendering was confirmed live in the TUI, not just inferred from the header being correct.
    - No `ABANDONED` exchanges anywhere.
  - **An open question, not resolved here**: the server row's kept-byte count landed at exactly 4096 B for both medium and large — matching the sendfile kprobe's (#68) own per-event cap exactly, not `512 (write prefix) + 4096 (kprobe sample) = 4608` as the syscall-shape numbers above would suggest. That 4608 figure *is* what the client row shows instead. Whether the write's 512 B prefix is being dropped rather than accumulated into the server-side body sample wasn't investigated further — noting it here rather than asserting a cause. The client row's own kept-byte count (4608 for medium, 3908 for large) isn't a fixed cap either; it comes from however many `read`/`recvfrom` calls curl happened to make, each individually capped at 4096 B, so it's expected to vary run to run rather than land on a round number.
  - **x86_64**: not tested — no x86_64 host was available in this session. `internal/loader/load.go` unconditionally skips attaching the sendfile kprobe when `runtime.GOARCH != "arm64"` (#112 tracks adding it), so the *sendfile* portion of medium/large should be entirely uncaptured there, but this is a reading of the code, not something run and observed.
  - **Net effect**: none of `writev`, `sendmsg`, or chunked `Transfer-Encoding` appear anywhere in this server's default paths, so #111, #113, #116, and #122 are all non-issues for Go's `net/http` — the only capture gap here is the sendfile path itself (#36's cap plus #112's still-unverified x86_64 kprobe). No `TCP_CORK`/`MSG_MORE` — only the usual `TCP_NODELAY` + keepalive `setsockopt`s.
- **Node.js `http.createServer` (#43)**: confirmed with `strace -f -tt -e trace=write,sendto,sendmsg,writev,sendfile,recvmsg,close` and the real `tinytap` TUI (driven non-interactively via `tmux send-keys`/`capture-pane`, same method as #42) against a `fs.createReadStream(file).pipe(res)` file-serving endpoint plus an inline `/hello` handler (`res.end('Hello, world')`), matching the issue's setup. Node never calls `sendfile` for the streaming path — file bytes go through the JS heap exactly as the issue predicted — so #68/#112 don't apply here at all.
  - **Syscall shape (strace)**: leaving `Content-Length` unset (the natural result of `.pipe(res)`) makes Node fall back to `Transfer-Encoding: chunked`. For all three of small.txt, medium.txt, and large.txt, the *entire* chunked response — status line, headers, chunk-size line, chunk data, and the chunk's trailing CRLF — goes out as a **single `writev` call with 4 iovecs**: `[headers + chunk-size line, "\r\n", chunk data, "\r\n"]` (363 B / 8357 B / 51365 B on the wire respectively). The terminating zero-length chunk (`"0\r\n\r\n"`) is then written as a **separate plain `write` syscall** once the stream ends — confirmed identically on all three sizes and on image.png. image.png produces the same 4-iovec `writev` shape (230 B on the wire) with the PNG bytes as the body iovec.
  - **Pairing (real TUI, not inferred)**: every request paired successfully with a 200 status — no `ABANDONED` anywhere, across small/medium/large/image — confirming #116's chunked-CRLF fix holds even though the terminator arrives in a syscall completely separate from the one carrying the chunk-size/body/CRLF. Detail-panel body byte counts, read from the actual TUI:
    - small.txt (200 B body): **200/200 B**, not truncated, on the server row.
    - medium.txt (8192 B real body) and large.txt (51200 B real body): **1024/1024 B — truncated**, identically, on the server row, despite the two real bodies differing by 43× — re-verified after #128 (was 256/256 B under the old front-loaded schedule; see the "Update" note below).
    - image.png (68 B, `Content-Type: image/png`): **`Response body: [image/png, 68 bytes]`** — the #117 binary placeholder rendered correctly.
    - The curl client row told a different story: **3933/3933 B — truncated** for *both* medium and large (vs. 1024 B on the server row), and unsurprisingly 200/200 and the image placeholder matched the server row for the other two. Unchanged by #128 — the client's `read`/`recvfrom` cap is a separate single-buffer path, not touched by `iov_sample_budget`.
    - The "N/N" shape above (kept == the second number) is not a typo and doesn't mean untruncated: per `internal/protocols/http/pairer.go`'s `bodyBytes`, the second number is `Content-Length` when the response declares one, but for a chunked response with no `Content-Length` — every case here — it falls back to `len(BodySample)`, i.e. exactly what tinytap managed to capture. So the "total" column is *itself* the captured-byte count, not the real 8192 B / 51200 B wire size; the `— truncated` suffix (a separate boolean) is the only signal that more was actually sent than shown.
  - **Update (#128) — why 1024 B, not 4096 B**: `fill_iov_payload` (bpf/tinytap.bpf.c) gives each iovec its own sampling budget rather than one shared 4096 B pool. This was originally front-loaded onto iovec 0 (2816/512/256/128 B for iovecs 0–3), which crushed this exact case: Node's chunk data always lands in iovec 2, so under that schedule any chunked body larger than 256 B was truncated at exactly 256 B no matter how large the real body was — a materially lower, size-independent ceiling than the flat 4096 B single-buffer cap that governs Python's/Go's plain `sendto`/`write` bodies (#41/#42). #128 replaced the front-loaded split with a flat **1024 B per iovec across iovec 0–3** (re-verified here with the same `strace`+TUI method): medium.txt and large.txt both now show **1024/1024 B** kept on the server row, a 4x improvement, still stopping after iovec 2 the moment it exceeds its (now larger) budget. small.txt (200 B) and image.png (68 B) both fit comfortably under the budget either way, hence full visibility in both schedules. The client row's 3933 B figure is consistent with a single `recvfrom`/`read` hitting close to the flat 4096 B cap (4096 minus roughly the 163 B of header/chunk-size/CRLF bytes the parser consumes before counting body) — packet-boundary dependent rather than a fixed constant, but consistently much larger than the server side's 1024 B, a clear illustration of why tinytap pairs two independently-captured views of the same exchange instead of assuming symmetric visibility.
  - **The inline `/hello` endpoint** contrasts with the streaming path: `res.end('Hello, world')` makes Node auto-compute `Content-Length: 12` and collapse the whole response into a single `writev` with iovec 0 carrying all 135 B (status line + headers + body) and an empty second iovec — well within budget, fully visible, no chunking. Whether a Node response hits the iovec-budget ceiling above or the flat cap depends entirely on whether the handler sets `Content-Length` before the first write, not on body size.
  - **Net effect**: #111 is why any of this pairs or parses at all (body lives in iovec 2, not iovec 0); #116 is why chunked framing doesn't get `ABANDONED` despite the terminator arriving via a wholly separate `write` syscall; #113 (sendmsg/recvmsg) and #112 (x86_64 sendfile kprobe) don't apply — Node's stream-pipe path never touches either. #128 (this recheck) rebalanced the writev-iovec budget from front-loaded to flat, roughly quadrupling how much of a chunked body is visible here — Axum (#104, "writev expected") is the next server likely to land in the same iovec-budget territory once verified.
- **nginx (#44)**: exercised on this Lima VM (arm64, nginx/1.28.0) across three configurations — static with `sendfile on` (the default), static with `sendfile off`, and reverse proxy (`proxy_pass` to a `python3 -m http.server` backend) — all with `tcp_nopush off`. Each was verified with `strace -f -tt -e trace=write,writev,sendto,sendmsg,sendfile,recvfrom,recvmsg` on the nginx worker plus the real `tinytap` TUI (driven non-interactively via `tmux send-keys`/`capture-pane`), for all four body sizes. No `ABANDONED` exchanges anywhere across any of the 12 request/response pairs; nginx never used chunked `Transfer-Encoding` in any of these tests since `Content-Length` is always known ahead of time for static files and buffered proxy responses, so #116/#122 don't apply here.
  - **`sendfile on` (default)**: headers go out via a single-iovec `writev`, then the entire body goes out via exactly one `sendfile` call regardless of size — same shape as Go's sendfile fast path. Body visibility is capped at a flat 4096 B by the arm64 kprobe (#68) since headers and body are on separate syscalls with independent budgets: small ✅ 200/200, medium ⚠️ 4096/8192, large ⚠️ 4096/51200.
  - **`sendfile off`**: nginx coalesces headers and body into a single `writev` call with the body starting at `iovec[1]` (small/medium) or spanning two separate `writev` calls when the body is large enough to hit nginx's own internal buffer split (large.txt: 32768 B + 18432 B). This is exactly the scenario #111's multi-iovec fix targets, and it does make the body visible — but the *amount* visible is below the flat 4096 B cap because `bpf/tinytap.bpf.c`'s `iov_sample_budget()` gives each iovec index its own small budget rather than one shared pool. **Update (#128)**: the budget schedule was originally front-loaded onto `iovec[0]` (2816/512/256/128/128/128/64/64 B), which crushed this exact case — medium.txt's single 2-iovec `writev` (header 241 B + body 8192 B) only got 512 B of body visible. #128 replaced that with a flat 1024 B per iovec across `iovec[0..3]` (0 B beyond), re-verified here: medium.txt now shows **1024/8192 visible** (header fits `iovec[0]`, body gets the full 1024 B `iovec[1]` budget). large.txt's two independent `writev` events each restart the budget schedule: the first (header + 32768 B body, 2 iovecs) contributes 1024 B of body; the second (18432 B body, 1 iovec, no header) gets the full `iovec[0]` budget of 1024 B → **1024 + 1024 = 2048/51200 visible** (down from the old schedule's 3328 B, a deliberate trade — see #128 for why). small.txt (438 B total) and image.png (68 B body) fit entirely within `iovec[0]`+`iovec[1]`'s combined budget, so both are fully visible/placeholder-rendered, unaffected by the schedule change.
  - **reverse proxy**: `sendfile` is never invoked regardless of the static-side `sendfile` setting — nginx's `proxy_pass` path reads the upstream response into its own proxy buffer (default ~4 KiB chunks) and re-`writev`s it downstream, issuing **many** `writev` calls per response once the body exceeds one buffer's worth (large.txt took 6 separate `writev` calls, 1–4 iovecs each). Each `writev` event independently restarts the same fixed budget schedule as the `sendfile off` case, so more (smaller) `writev` calls means more total sampled bytes. **Update (#128)**: with the flat 1024 B/iovec schedule, medium.txt (one 4-iovec `writev`: header + 3 body chunks) → **1024/8192 visible** (up from 512 B under the old schedule, since sampling still stops after `iovec[1]`'s budget is exhausted regardless of how many iovecs follow, but that budget is now 1024 B instead of 512 B). large.txt spreads across 6 `writev` calls (one header+body, five pure-body) → 1024 + 1024×5 = **6144/51200 visible** (down from the old schedule's 14592 B, since the old `iovec[0]` budget of 2816 B on the five pure-body calls is now only 1024 B each — the same trade discussed in #128: no schedule can be front-loaded for every server's iovec layout without starving some other server, so this now lands in a narrower, less surprising band alongside the other rows instead of an outlier high). tinytap also captures nginx's own outgoing request to the backend as a second, independent exchange (nginx acting as an HTTP/1.0 client) — worth noting as a two-hop capture artifact of proxying, not a bug.
  - **Not tested this pass**: `tcp_nopush on` (TCP_CORK behavior) was not exercised — the scaffold supports toggling it but only `off` was run given time constraints; a future pass could confirm whether it changes the iovec/writev shape at all.
  - **Net effect**: the sub-issue's original premise — "sendfile blocks static visibility, but reverse-proxy is visible" — doesn't hold as a clean dichotomy. All three configurations are partially visible, and which one shows *more* bytes depends on how many separate `writev`/`sendfile` syscalls the body happens to be split across, not on sendfile-vs-proxy as such. The one clean rule that does hold: `sendfile`'s body is invisible unless the #68 kprobe covers it (arm64 only, #112 open for x86_64), while any `writev`-based path is visible via #111's fix, just possibly only a small fraction of the body.
